14.2. Luồng request qua Interceptor

🟢 Mọi request đều đi qua 2 interceptor trong src/api/httpClient.js — không bypass được.

Interceptor REQUEST (trước khi gửi):

  1. Service gọi apiClient.get('/news') hoặc bất kỳ method nào.
  2. Interceptor kiểm tra URL có khớp /auth/login hoặc /auth/refresh không (SKIP_AUTH_PATTERNS).
  3. Nếu token gần hết hạn theo VITE_AUTH_REFRESH_BEFORE_EXPIRES_MS → gọi authTokenRefresh.js để refresh trước khi gửi request.
  4. Nếu không skip → gọi getAuthorizationHeader() từ authSession.js.
  5. hasValidAccessToken() kiểm tra token tồn tại và chưa hết hạn (Date.now() < expiresAt).
  6. Nếu có token hợp lệ → tự động thêm header Authorization: Bearer <token>.
  7. Nếu VITE_API_CSRF_ENABLED=true và method là POST/PUT/PATCH/DELETE → gắn CSRF token từ meta/cookie vào header.
  8. Request bay đến backend.

Interceptor RESPONSE (sau khi nhận):

  1. Response trả về → kiểm tra HTTP status.
  2. Nếu status 401 và còn refresh token/cookie refresh → refresh token rồi retry request cũ đúng 1 lần.
  3. Nếu refresh thất bại hoặc đã retry rồi vẫn 401 → gọi clearAuthSession() + notifyUnauthorized().
  4. MainLayout.vue lắng nghe event notifyUnauthorized → hiện banner "phiên đăng nhập hết hạn".
  5. Nếu status 0, 403, 500 → gọi logTechnicalError() (chỉ log ở DEV mode).
  6. Lỗi được normalize qua normalizeAppError() → trả ra message i18n thân thiện.
  7. Store bắt lỗi, lưu vào error.home / error.list / error.detail…
  8. Page hiển thị thông báo lỗi cho user.

14.4. Mock vs Real API — công tắc chuyển đổi

  • Service dùng data từ src/data/news.js và src/data/categories.js.
  • Có delay(250ms) giả lập latency.
  • Không cần backend chạy.
  • Axios instance vẫn tồn tại nhưng không gọi HTTP.
  • Dùng để dev UI, demo, test offline.
🟢 REAL API MODE — VITE_USE_MOCK_API=false
  • Service gọi apiClient.get('/news') thật.
  • Nếu backend trả lỗi hoặc data rỗng → tự fallback về mock data.
  • Token auth tự gắn vào header qua interceptor.
  • Cache TTL theo VITE_NEWS_CACHE_TTL_MS (mặc định 60s).
  • Cần backend đang chạy tại VITE_API_BASE_URL.

Chuyển đổi: chỉ cần sửa .env, không sửa code:

14.5. 🔴 Bảo mật API

🔴 BẢO MẬT — Token lưu trong memory JavaScript (không phải localStorage/cookie), tự xóa khi 401.

Cơ chế bảo mật hiện tại:

  1. Token chỉ lưu in-memory phía browser — authSession.js không lưu localStorage, không lưu cookie, và không lưu token vào module state khi chạy SSR server. Mất khi F5 hoặc đóng tab nếu không dùng cookie/httpOnly refresh.
  2. Kiểm tra expiry trước khi gắn token — hasValidAccessToken() so sánh Date.now() < expiresAt. Nếu token gần hết hạn thì request interceptor tự refresh trước theo VITE_AUTH_REFRESH_BEFORE_EXPIRES_MS.
  3. SKIP_AUTH_PATTERNS bảo vệ endpoint login/refresh — /auth/login và /auth/refresh bị loại khỏi việc gắn token (tránh vòng lặp và leak token cũ).
  4. skipAuth flag — khi gọi apiClient.post(endpoint, data, { skipAuth: true }) thì interceptor bỏ qua gắn token. Dùng cho login, refresh.
  5. Auto refresh và retry 401 một lần — nếu backend trả 401 và còn refresh token/cookie refresh, interceptor gọi authTokenRefresh.js, gắn token mới và retry request cũ đúng 1 lần.
  6. Auto clear khi refresh thất bại — nếu refresh fail hoặc retry vẫn 401, interceptor gọi clearAuthSession() và notifyUnauthorized(). Không để token cũ tồn tại.
  7. Chống refresh storm — nhiều request cùng lúc chỉ dùng chung một promise refresh, tránh bắn nhiều request /auth/refresh song song.
  8. CSRF header tùy chọn — khi VITE_API_CSRF_ENABLED=true, request ghi dữ liệu tự gắn CSRF token từ meta/cookie vào header cấu hình bởi VITE_API_CSRF_HEADER_NAME.
  9. notifyUnauthorized listener — MainLayout.vue đăng ký listener để bắt event 401, hiển thị banner thông báo hết phiên mà không redirect cứng.
  10. DOMPurify sanitize HTML — nội dung bài viết dùng v-html nhưng bắt buộc qua sanitizeHtml.js (DOMPurify) trước khi render. Ngăn XSS từ nội dung CMS.
  11. Nuxt middleware sanitize slug/query — middleware/route-sanitizer.global.js sanitize params slug và query string trước khi cho vào component. Ngăn path traversal và injection qua URL.
  12. normalizeSearchInput() — search query bị strip HTML và whitespace trước khi dùng. Ngăn XSS qua tìm kiếm.
  13. logTechnicalError chỉ chạy DEV — if (import.meta.dev || import.meta.env?.DEV) mới console.error. Production không lộ stack trace ra console.
  14. X-Requested-With header — Axios instance gắn mặc định X-Requested-With: XMLHttpRequest. Giúp backend phân biệt AJAX request và block CSRF từ form HTML thông thường.
  15. .env không commit — .gitignore bỏ .env. Secret key, API URL, token config không vào repo.
🟡 Điểm cần lưu ý thêm khi mở rộng bảo mật:
  • Nếu backend dùng cookie httpOnly: bật VITE_API_WITH_CREDENTIALS=true và cấu hình CORS backend cho phép credentials.
  • Nếu backend refresh bằng cookie httpOnly thay vì refresh token trong response body: bật VITE_AUTH_REFRESH_USES_COOKIE=true.
  • Nếu bật CSRF: backend cần phát cookie/meta token không httpOnly cho frontend đọc, và kiểm tra header tương ứng ở các method ghi dữ liệu.
  • API /logs nếu có: nên dùng skipAuth: false (gắn token) để backend biết user nào gửi log.
  • API /information nếu public: không cần token, interceptor tự bỏ qua khi không có token trong session.
  • Không bao giờ log accessToken hay credentials ra console, ngay cả trong DEV.

Quan trọng: API kết nối qua một cửa duy nhất

Project hiện dùng src/api/httpClient.js làm Axios instance trung tâm. Các service như newsService.js, categoryService.js, authService.js không tự tạo Axios mới mà import apiClient rồi gọi endpoint tương ứng.

  • src/config/apiConfig.js đọc VITE_API_BASE_URL, VITE_API_TIMEOUT, VITE_API_WITH_CREDENTIALS và các biến CSRF.
  • src/api/httpClient.js gắn baseURL, timeout, header JSON, auth header, CSRF header, auto refresh token và retry 401 một lần.
  • src/services/*.js là nơi viết từng nhóm API nghiệp vụ: news, category, information, log, contact...
  • src/stores/*.js gọi service, lưu loading/error/cache/state để page dùng.
Không nên gọi API trực tiếp trong component

Component hoặc page chỉ nên gọi store/action. Nếu sau này có nhiều API như category, information, log, contact, admin... thì tạo service riêng cho từng nhóm, mapper riêng nếu payload phức tạp, rồi cho store gọi service. Cách này giúp đổi endpoint/backend mà ít ảnh hưởng giao diện.

Bảo mật API khi thêm nhiều endpoint
  • Không đưa secret, private key, database password, API key nội bộ vào .env frontend. Biến bắt đầu bằng VITE_ sẽ có thể xuất hiện trong bundle browser.
  • Frontend chỉ giữ URL public, timeout, flag mock, endpoint public. Secret phải nằm ở backend.
  • Mọi dữ liệu nhập từ user cần validate ở frontend để UX tốt, nhưng backend vẫn phải validate lại vì frontend có thể bị bypass.
  • Nếu dùng cookie auth và VITE_API_WITH_CREDENTIALS=true, backend phải cấu hình CORS đúng domain, không dùng wildcard * cho credential.
  • Nếu dùng CSRF, bật VITE_API_CSRF_ENABLED=true và thống nhất cookie/header CSRF với backend.
  • Nếu dùng Bearer token, hạn chế lưu token ở localStorage. Project hiện lưu token in-memory trong authSession.js, an toàn hơn trước XSS lưu trữ nhưng reload sẽ mất phiên.
  • Không log dữ liệu nhạy cảm: token, password, refresh token, cookie, email cá nhân, số điện thoại, payload form đầy đủ, response chứa thông tin riêng tư.
  • Endpoint log nên giới hạn kích thước payload, rate limit, loại bỏ field nhạy cảm và chỉ nhận thông tin cần debug.

VITE_API_BASE_URL là domain gốc của backend, ví dụ https://api.example.com/api. Khi service gọi apiClient.get('/categories'), request thật sẽ thành https://api.example.com/api/categories.

Checklist trước khi bật API production
  • Kiểm tra VITE_USE_MOCK_API=false để app không còn dùng mock data.
  • Kiểm tra VITE_API_ALLOW_MOCK_FALLBACK=false để API lỗi không âm thầm hiện dữ liệu demo.
  • Kiểm tra VITE_API_BASE_URL đúng domain HTTPS production.
  • Kiểm tra backend trả JSON shape đúng mapper đang normalize.
  • Kiểm tra CORS chỉ cho phép domain frontend thật.
  • Nếu dùng cookie auth: bật VITE_API_WITH_CREDENTIALS=true, cấu hình CORS credential, và cân nhắc VITE_AUTH_REFRESH_USES_COOKIE=true.
  • Nếu backend yêu cầu CSRF: bật VITE_API_CSRF_ENABLED=true và thống nhất cookie/header CSRF.
  • Kiểm tra endpoint log không nhận dữ liệu nhạy cảm và có rate limit.
  • Kiểm tra lỗi 401 có auto refresh/retry một lần; nếu refresh fail thì clear session và hiển thị thông báo phù hợp.
  • Không commit file .env thật lên GitHub.

Database trong sơ đồ trên là giả định ở backend. Repo frontend hiện tại chưa có database.
